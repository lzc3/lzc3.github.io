Redis数据结构及类型

基础数据类型

SDS

  • 为什么不用c自带的字符串？

​ 其本质是字符数组，获取长度需要遍历、以\0作为结束符，不安全，不能存储图片、音频…

Sds优势

  • 长度获取速度为O(1)
  • 动态扩容
  • 减少内存分配次数(扩容的时候内存预分配，一版扩大为原来的两倍+1.大于1M的情况下就是 + 1M + 1)
  • 二进制安全

组成：

  • len 说明当前字符串大小
  • alloc 申请的内存空间
  • flag 标识是那种动态字符串(长度为 255 ..)
  • 字符数组

IntSet

​ Set的一种实现，基于整数数组来实现，并且具备长度可变、有序的特征

连续内存，寻址方便

超出当前编码范围，全部使用新的编码

特性

  • redis保证其元素唯一、有序
  • 类型升级，可节省空间
  • 底层使用二分查找来查询

Dcit

扩容

负载因子 >= 1，并且服务器没有执行bgsave或者bgweiteaof等后台进程

负载因子 > 5

ZipList

连锁更新问题

  • 申请的内存是连续空间(因此要限制其大小)

  • 压缩列表可以看作一种连续内存空间的双向链表

  • 列表的节点之间不是通过指针连接，而是记录上一节点和本节点的长度来寻址，内存占用较低

  • 如果列表数据过多，导致长度过长，可能影响查询性能

  • 增或删较大数据的时候有可能发生连续更新问题

QuickList

为了避免QuickList中每个ZipList的entry过多，Redis提供了一个配置项：list-max-ziplist-size

  • 值为正，代表ziplist的允许的entry个数的最大值
  • 值为负，代表ziplist的最大内存大小，默认值为-2，代表每个zipList不能超过8kb

除了控制ZipList的大小，QuickList还可以对节点的ZipList做压缩

因为链表一般都是从首尾访问较多，所以首尾是不压缩的，这个参数是控制首尾不压缩的节点个数：

  • 0 不压缩 默认

  • 1首尾各有1个节点不压缩

  • 2首尾各有2个节点不压缩

    …

  • 是一个节点为ZipList的双端链表

  • 节点采用ZipList，解决了传统链表的内存占用问题

  • 控制了ZipList大小，解决了连续内存空间申请效率问题

  • 中间节点可以压缩，进一步节省了内存

SkipList

  • 双向链表，每个节点包含score和ele值
  • 节点按照score排序，score值一样按照ele字典排序
  • 每个节点都可以包含多层指针，层数是1~32之间的随机数
  • 不同层指针到下一节点的跨度不同，层级越高，跨度越大
  • 增删改查效率于红黑树基本一致，实现却更简单

RedisObject

​ Redis中的任意数据类型的键和值都会被封装成一个RedisObject，也叫做Redis对象。

基础数据类型

String

  • 基本编码为RAW，基于SDS实现，存储上限为512mb
  • 如果存储的SDS长度小于44字节，则会采用EMBSTR编码，此时object head于SDS是一段连续的空间，只需要调用一次内存分配函数
  • 如果存储的字符串是整数值，并且大小在LONG_MAX范围内，则会采用INT编码，直接将数据保存在RedisObject的ptr指针位置(刚好8字节)，不需要SDS了

List

可以使用的数据结构

  • LinkedList
  • ZipList
  • QuickList

3.2版本前，使用LinkedList和ZipList实现，之后使用QuickList

Set

  • 不保证有序性
  • 保证元素唯一
  • 交集、并集、差集

可以使用的数据结构

  • Dict key用来存储元素，value统一为null
  • 当存储的所有数据都是整数，并且元素数量不超过set-max-intset-entries时，会采用IntSet编码，已节省内存

ZSet

  • member必须唯一
  • 可以根据member查询值
  • 可以根据score排序

其数据结构必须满足键值存储、键必须唯一、可以排序

  • SkipList满足了排序和存储键值
  • HT(Dict)：可以键值存储，并且可以根据key查找value

当元素不多的时候，HT和skipList的优势不明显，而且更占用内存，因此zset还会采用ziplist结构来节省内存，不过需要满足两个条件

  • 元素数量小与zset_max_ziplist_entries 默认128
  • 每个元素都小于zset_max_ziplist_value字节，默认64

Hash

Hash结构与Redis中的Zset非常类似

  • 都是键值存储
  • 都需要根据键获取值
  • 键必须唯一

不同点：

Zset的值时score，需要排序
